## Codebase Patterns
- Server typechecks with `npx tsc --noEmit -p tsconfig.server.json` (separate from frontend tsconfig)
- Server code uses `.js` extensions in imports despite being `.ts` files (ESM convention)
- Server runs via `tsx` — no compilation step needed
- DB schema uses drizzle-orm with SQLite (better-sqlite3)
- Existing agent DB schema in `server/db/schema/agents.ts` has status enum: idle, working, paused, error
- Pipeline stages have status enum: queued, running, done, needs-you, error
- WebSocket broadcast utility in `server/ws/handler.ts` — use `broadcast(event, data)`
- Agent adapters use lazy process init — constructor takes config, process spawns on first `sendMessage()`
- Agent adapters use a pending message queue + resolver pattern for async response consumption
- MCP adapter uses JSON-RPC 2.0 protocol — requests have `id` (get response), notifications have no `id`
- MCP handshake: initialize → initialized notification → ready for tool calls
- Factory `createAgent()` is async — returns `Promise<AgentAdapter>` because 'auto' mode probes MCP
- Settings table is workspace-scoped: query with `and(eq(workspaceId), eq(key))` pattern
- Use `getTransportSetting(workspaceId)` to read transport preference from DB, defaults to 'auto'
- PipelineEngine: instantiate with `new PipelineEngine(pipelineId)`, then call `await engine.run()`
- Use `asc()` / `desc()` from `drizzle-orm` for orderBy clauses
- Use `typeof table.$inferSelect` to get the row type from a drizzle schema table
- Adding enum values to drizzle SQLite text columns only changes TS types — no DB migration needed
- Parallel stages: `groupStagesBySortOrder()` groups by sortOrder, `executeStageGroup()` runs groups via `Promise.allSettled`
- Checkpointing is at group level (in `executeStageGroup`), not individual stage level
